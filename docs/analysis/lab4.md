## Аналіз

### 1. Що змінилося порівняно з лабораторною №3?

У Лабораторній №3 основний акцент був зроблений на Command Query Separation, де вся бізнес-логіка була зосереджена в хендлерах, що виконувалися послідовно в межах однієї транзакції.
У Лабораторній №4 структура змінилась таким чином:

1. **Відокремлення побічних ефектів**: було винесено нотифікації в окремий допоміжний компонент, який має власний інтерфейс.  
2. **Впровадження Event Bus**: з'явився шар інфраструктури для публікації та підписки на події, що дозволило відв'язати основні команди від допоміжних операцій.  

### 2. Побічні операції

Було виділено такі операції:

* **Надсилання Email-сповіщення**: вибрано як основну побічну операцію при створенні замовлення та акаунту.
* **Обґрунтування**: надсилання листа не є критичним для успішного збереження замовлення в БД. Це типове завдання, затримка якого не повинна впливати на досвід користувача.

### 3. Порівняння синхронного та асинхронного підходів

* Синхронний підхід вирізняється високим часом відповіді API, оскільки загальна затримка дорівнює сумі часу виконання бізнес-логіки та часу очікування відповіді від допоміжного сервісу. Зв'язаність між компонентами при цьому є високою, оскільки відправник повинен знати адресу, інтерфейс та логіку роботи отримувача. Складність реалізації синхронного підходу є досить низькою через послідовність виконання та наявність єдиного потоку виконання.  

* Натомість асинхронний підхід дозволяє суттєво зменшити час відповіді API, оскільки клієнт отримує підтвердження виконання основної операції миттєво, не чекаючи завершення фонових задач. Рівень зв'язаності між компонентами мінімальний - відправник лише публікує подію і не знає про існування конкретних отримувачів. Проте цей підхід супроводжується вищою складністю реалізації, яка включає налаштування шини подій та обробку помилок у фонових процесах.  

### 4. Вибір для Production-системи

Для production-системи краще підходить асинхронний підхід. Хоча він потребує складнішої інфраструктури, він забезпечує необхідну масштабованість та відмовостійкість. Основна бізнес-транзакція стає незалежною від доступності зовнішніх сервісів, що критично для сучасних високонавантажених систем.

### 5. Ідемпотентність та обробка дублікатів

В асинхронних системах існує ризик отримання однієї події двічі.

* **Що станеться**: якщо не обробити дублікат, клієнт отримає два однакових листа про створення замовлення.
* **Ідемпотентність обробників**: мій обробник для простоти додатку не має спеціальної обробки дублікатів.
