## Аналіз архітектурних змін

### 1. Що змінилося в структурі проєкту

Порівняно з другою лабораторною роботою, у лабораторній 3 відбулася декомпозиція на рівні виконання операцій. Великі хендлери, які обробляли бізнес-операції, були розділені на маленькі операції запису, або операції читання.

### 2. Переваги CQS: чим допомогло розділення

* **Ізоляція контексту**: Розділення на команди та запити дозволило оптимізувати кожен шлях окремо.
* **Одинична відповідальність**: Кожен хендлер займається лише однією дією. Це полегшує розробку, оскільки полегшується орієнтація в коді.
* **Тестування**: Тепер можна тестувати бізнес-логіку окремо від інфраструктури, використовуючи `FakeUserRepository` та моки, що прискорює тестування.

### 3. Недоліки CQS: що ускладнилося

* **Збільшення кількості файлів**: Для найпростішої операції потрібно створити команду, хендлер. Це створює невеликий архітектурний оверхед.
* **Складність навігації**: Потрібно більше часу, щоб зрозуміти повний ланцюжок викликів, оскільки код розкиданий по різних шарах.

### 4. Command / Query Handler vs Service

* **Service**: Має доступ до всього, порушує принцип єдиної відповідальності, важкий для рефакторингу. Зміна однієї логіки може випадково зламати іншу функціональність у тому ж сервісі.
* **Command/Query Handler**: Кожна дія інкапсульована. Сервіс - це процедурний підхід, тоді як хендлери - це об'єктно-орієнтований/архітектурний підхід, де логіка чітко розділена.

### 5. Розширюваність

* **Додавання нової команди**: Щоб додати нову команду треба створити новий клас для цієї команди, та новий хендлер. Ми нічого не змінюємо в існуючих хендлерах, що відповідає принципу Open/Closed Principle.
* **Зміна існуючого сервісу**: У сервісі нам довелося б редагувати існуючий метод, ризикуючи створити помилки. CQS робить розширення безпечним.

### 6. Query data vs Domain model

Структура даних, яку повертає Query, відрізняється від доменної моделі.

* **Доменна модель** - це внутрішня структура з бізнес-правилами, методами та інваріантами.
* **Query DTO** - це структура, оптимізована для видачі клієнту (API).  

Це дозволяє змінювати базу даних або доменну модель, не змінюючи API-відповіді для клієнта.
