# Порівняльний аналіз

## 1. Структурні зміни в проєкті порівняно з Лабораторною роботою №1

У першій лабораторній роботі проєкт мав таку структуру, де логіка Flask, обробка HTTP-запитів, бізнес-правила та робота з базою даних за допомогою SQLAlchemy були жорстко переплетені в межах контролерів.

В результаті рефакторингу структура проєкту була повністю реорганізована та розподілена на чотири ізольовані шари:

1. **Domain Layer (Шар домену)**: `src/domain/models.py`, `factories/`, `errors.py`. Містить виключно чисту бізнес-логіку, сутності у вигляді стандартних Python-класів, фабрики для збирання агрегатів та доменні виключення. Цей шар не має жодних зовнішніх залежностей.
2. **Application Layer (Шар застосунку)**: `src/application/handlers.py`, `dto.py`. Визначає сценарії використання системи (Use Cases). Через хендлери координує виконання бізнес-операцій, оперує інтерфейсами репозиторіїв та Unit of Work, не знаючи про конкретну реалізацію БД.
3. **Infrastructure Layer (Шар інфраструктури)**: `src/infrastructure/models.py`, `repositories.py`. Містить конкретну технічну реалізацію: SQLAlchemy-моделі (Entities для БД), конфігурацію підключення до бази даних та реалізацію паттернів Repository і Unit of Work.
4. **Presentation Layer (Шар представлення)**: `src/presentation/controllers.py`. Відповідає за HTTP-інтерфейс, маршрутизацію Flask, валідацію вхідних JSON-пакетів та керування сесіями автентифікації за допомогою JWT-токенів.

## 2. Переваги розділення проєкту на шари

Впровадження шарової архітектури дозволило вирішити ключові проблеми масштабованості та супроводу коду:

* **Висока тестованість**: Завдяки ізоляції домену та застосунку, була отримана можливість покрити систему Unit-тестами (`tests/domain` та `tests/application`) без підняття бази даних чи сервера. Весь шар застосунку тестується за мілісекунди завдяки підміні інфраструктури фейковими об'єктами.
* **Слабка зв'язаність**: Компоненти взаємодіють через абстракції. Презентаційний шар нічого не знає про логіку збереження в БД, він лише викликає відповідний хендлер.
* **Дотримання Single Responsibility Principle**: Тепер кожен клас виконує лише одну задачу. Наприклад, `UserRepository` відповідає виключно за SQL-запити до таблиці користувачів, `User` (домен) - за валідацію правил імені користувача, а `UserHandler` - за оркестрацію процесу реєстрації.
* **Паралельна розробка**: Розділення інтерфейсів дозволяє різним розробникам одночасно працювати над API-ендпоінтами та оптимізацією SQL-запитів у репозиторіях, не блокуючи один одного.

---

## 3. Недоліки та ускладнення, що з'явилися

Попри очевидні переваги, така архітектура принесла з собою певні проблеми:

* **Суттєве збільшення обсягу коду**: Проста операція, яка раніше займала 5 рядків у контролері Flask, тепер вимагає створення DTO для вхідних даних, методу в хендлері, методу в інтерфейсі репозиторію, його реалізації в інфраструктурному репозиторії та мапінгу об'єктів.
* **Необхідність мапінгу**: Поява двох паралельних моделей для кожної сутності (чиста доменна модель `Product` та інфраструктурна `ProductEntity` для SQLAlchemy) змусила створювати окремі мапери. Перетворення об'єктів туди-сюди при кожному запиті до БД створює додаткове навантаження на процесор та збільшує ймовірність помилок узгодження полів.
* **Складніша навігація**: Для відстеження логіки виконання одного запиту доводиться стрибати між 4–5 різними файлами у різних каталогах системи. Це також підвищує поріг входження для нових програмістів у проєкт.

## 4. Оцінка простоти зміни бази даних або веб-фреймворку

Розділення на шари покращило гнучкість системи щодо заміни технологічного стеку:

### Зміна бази даних

Завдяки *Repository*, шар застосунку та домену повністю ізольований від типу БД. Наприклад, перехід з `PostgreSQL` на `MongoDB` вимагатиме лише написання нових інфраструктурних класів репозиторіїв, які реалізують базовий інтерфейс `AbstractProductRepository`. Жоден рядок бізнес-логіки чи код валідації в домені при цьому не зміняться.

### Зміна Веб-фреймворку

Якщо виникне потреба замінити `Flask` на `FastAPI` або `Django Ninja` для покращення асинхронності, зміни торкнуться виключно шару представлення (`src/presentation/controllers.py`). Нам знадобиться лише переписати декоратори маршрутів та систему витягування токенів під новий фреймворк. Вся внутрішня логіка, хендлери та взаємодія з базами даних залишаться недоторканими, оскільки вони не мають жодного імпорту із бібліотеки `flask`.

## 5. Обґрунтування вибору моделі домену

У даному проєкті було обрано **Anemic Domain Model** у поєднанні з **Application Handlers (Сервісним шаром)**.

### Особливості обраного підходу:

Доменні класи (наприклад, `User`, `Product`) реалізовані як чисті структури даних, які містять лише властивості та базову валідацію в методі `__post_init__`. Вся основна бізнес-логіка, координація транзакцій, генерація ідентифікаторів та збереження стану винесені у сервіси/хендлери.

### Причини вибору:

1. **Специфіка Python та SQLAlchemy**: В Python використання Rich Domain Model (де сутності самі містять методи роботи з базою даних) часто призводить до паттерну Active Record. Це порушує принципи Clean Architecture, адже доменні моделі стають жорстко прив'язаними до ORM-механізмів. Анемічна модель дозволила легко розділити `User` та `UserEntity`.
2. **Простота CRUD-операцій**: Оскільки застосунок на даному етапі має відносно лінійну бізнес-логіку, Rich Domain Model створила б надлишкове ускладнення. Анемічна модель є більш інтуїтивною для невеликих та середніх систем.
3. **Чіткий розподіл обов'язків**: Винесення логіки в `Handlers` дозволяє тримати сутності домену максимально легкими та зрозумілими. Фабрики відповідають за правильність збирання об'єктів, а хендлери - за послідовність кроків, що забезпечує гнучке керування транзакціями через Unit of Work.
